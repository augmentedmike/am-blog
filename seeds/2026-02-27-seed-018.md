## Hook

Most people use AI like a single-track railroad.

One task at a time. Wait for it to arrive. Send the next one.

That's fine. But a railroad's real power isn't one train — it's the *network*. Multiple trains, multiple tracks, a dispatcher watching the board, everything moving at once.

The mental model shift that changed how I work: **I stopped being a passenger and started being the dispatcher.**

---

## The Sequential Tax

Here's how most people run AI-assisted work:

1. Ask AI to build Feature A
2. Wait
3. Review, fix, iterate
4. Ask AI to build Feature B
5. Wait
6. Review, fix, iterate

Each task is sequential. The AI never waits, but *you* do — you're the bottleneck at every handoff.

A 6-hour build feels like 6 hours because you're in the loop for every step.

The dispatcher doesn't work that way.

---

## Parallel Tracks in Practice

The mechanism is simple: **git worktrees**.

A worktree is a separate checked-out copy of your repo at a different path. Each agent gets its own worktree — its own branch, its own filesystem, no conflicts with the others.

```bash
# Two agents, two tracks, same repo
git worktree add -b feature/auth ~/projects/agent-auth main
git worktree add -b feature/api  ~/projects/agent-api  main

# Launch the agents — stagger by 2 minutes to avoid rate limits
cd ~/projects/agent-auth && ralphy --codex --prd auth-prd.md &
sleep 120
cd ~/projects/agent-api  && ralphy --codex --prd api-prd.md  &
```

Now both agents are running. The auth system and the API layer are being built simultaneously, in isolation, on separate branches.

While they work, you're not waiting. You're writing the next PRD, reviewing something else, or asleep.

---

## What the Dispatcher Actually Does

Running parallel agents doesn't mean hands-off. It means your job *changes*.

**Before**: You were a taxi driver. Drive one passenger, drop them off, pick up the next.

**After**: You're a dispatch coordinator. You don't drive anything. You:

- Write clear PRDs (the routing instructions)
- Stagger launches to avoid rate-limit collisions
- Monitor heartbeats — is train 3 stuck at the wrong station?
- Merge the branches when each track arrives
- Resolve conflicts at the junction points

The work isn't gone. The nature of the work changes. You operate at a higher altitude.

---

## The Stagger Rule

One practical detail that bites people: **API rate limits**.

If you launch 3 Codex agents simultaneously, they all hit the API at once. You get 429s. Agents start failing or going slow.

The fix: space launches by 2 minutes minimum.

```bash
ralphy --codex --prd frontend-prd.md -C ~/projects/agent-frontend &
sleep 120
ralphy --codex --prd backend-prd.md  -C ~/projects/agent-backend  &
sleep 120
ralphy --codex --prd tests-prd.md    -C ~/projects/agent-tests    &
```

Three tracks, one dispatcher, 4 minutes of staggered starts. Then go do something else.

---

## Where Parallel Fails

Not everything parallelizes cleanly.

**Shared state** is the enemy. If both agents need to touch the database schema, you'll get merge conflicts that are painful to untangle. Rule: when in doubt, give each agent a domain boundary it owns completely — frontend vs. backend, auth vs. core logic, feature vs. tests.

**Architecture decisions** don't parallelize. If you're not sure how two pieces connect, build one first. Parallelizing on top of an unresolved architecture question means two agents that both need to be thrown away.

**Security-critical code** never gets delegated to parallel agents. Auth, payments, encryption — those get human attention, always, no matter how fast you want to ship.

The dispatcher's most important job isn't scheduling trains. It's knowing which tracks aren't ready yet.

## The Wake Hook

One more piece: **async completion**.

When an agent finishes, I don't want to check back an hour later to find it finished 55 minutes ago. That's waste.

The pattern:

```bash
tmux new -d -s agent-auth \
  "cd ~/projects/agent-auth && ralphy --codex --prd auth-prd.md; \
   EXIT_CODE=$?; \
   miniclaw system event \
     --text \"Auth agent done (exit $EXIT_CODE)\" \
     --mode now; \
   sleep 999999"
```

When the train arrives at the station, it fires a notification. The dispatcher knows immediately. No polling, no "I wonder if it's done yet" loops.

---

## What This Changes

Running 3 agents in parallel isn't 3x faster. It's more like 2.2x — coordination overhead is real. Merging branches takes time. Resolving the unexpected intersection points takes time.

But here's what actually changes: **your ceiling**.

With one sequential agent, a 6-hour build takes 6 hours of your attention.

With three parallel agents and good worktrees, a 6-hour build takes 3 hours of calendar time, 45 minutes of your attention, and you were probably sleeping for 5 of those hours.

That's not an incremental improvement. That's a different kind of work.
