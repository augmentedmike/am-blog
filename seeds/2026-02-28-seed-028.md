## Hook

Every AI conversation starts at zero.

No memory of last week's project. No memory of how you like your code structured. No memory of that customer call you told it about Tuesday. You explain yourself. Again. You restate your tech stack. Again. You remind it what your product actually does. Again.

It feels like hiring a brilliant intern who forgets your name every morning.

The problem isn't the model. The problem is that you're treating a stateless API call like a persistent relationship. Until you build the memory layer, the model is just a calculator — powerful, but amnesiac.

Here's how I fixed it.

## Layer 1: Working Memory — MEMORY.md

The first layer is the one that runs on every session. A single markdown file loaded into context at the start of every conversation. It contains:

- **Current priorities** — what's active, what's urgent, what's blocked
- **Hot facts** — key relationships, ongoing decisions, known constraints
- **Standing preferences** — how I like things structured, what I've already ruled out, what tools I use
- **Project roster** — what's alive, what's shipped, what's parked

This is the brain's RAM. Small, fast, always loaded. The AI reads it before it reads anything else.

What goes here: things that are true right now and matter to almost every conversation.

What doesn't go here: historical context, detailed specs, one-off facts. Those go deeper.

The discipline: keep it under 500 lines. If MEMORY.md bloats, the AI drowns in it. Every week, prune what's gone cold. Demote old facts to daily notes. Promote new priorities to the top.

---

## Layer 2: Episodic Memory — Daily Notes

The second layer is chronological. Every day, a dated markdown file captures what happened:

```
memory/2026-02-28.md
memory/2026-02-27.md
memory/2026-02-26.md
```

What goes here:
- What was built or shipped today
- Decisions made and why
- Conversations that generated new information
- Problems hit and how they were resolved
- Cron job and agent run logs

This is the brain's hippocampus — the episodic, time-stamped record. The AI can search it when it needs historical context: "What did we decide about the database schema last week?" It searches the daily notes, finds the entry, and answers without you having to remember.

The format matters: short, dense, factual. Not a journal. Not prose. Bullets and timestamps. Optimized for retrieval, not reading.

The discipline: entries get written automatically by cron jobs and agent logs. Human entries are brief additions. The system should write most of this layer itself.

## Layer 3: Semantic Memory — The Knowledge Graph

The third layer is structured, entity-based, and designed for long-term retention:

```
memory/life/
  people/
    mike.json
    clients/
  projects/
    bonsai/
    am-blog/
  preferences/
  tech-stack/
```

Each entity is a JSON file with typed facts, access tracking, and relationship pointers:

```json
{
  "id": "pref-001",
  "type": "preference",
  "domain": "code-style",
  "fact": "Prefers TypeScript over JavaScript for all new projects",
  "confidence": 0.95,
  "firstSeen": "2026-01-15",
  "lastAccessed": "2026-02-27",
  "accessCount": 14,
  "status": "active"
}
```

This is the brain's cortex — persistent, structured, updateable. The AI reads this layer when it needs to know something specific about a person, project, or preference. When facts change, old entries get superseded (never deleted). The history stays; the current truth is always findable.

The discipline: never delete facts. Supersede them. Add `"status": "superseded"` and `"supersededBy": "new-id"`. This way, if you ever need to understand how a decision evolved, the trail is intact.

## The Search Layer That Ties It Together

Three layers of memory are only useful if you can search them without thinking about which layer to check. This is what QMD handles:

```bash
qmd search "database schema decision"
```

It searches MEMORY.md, all daily notes, and all knowledge graph entities simultaneously. Returns the most semantically relevant snippets, ranked by recency and match quality. The AI uses this before answering any question that might have prior context — not as an afterthought, but as a mandatory step.

This is the part most people skip. They have memory files but no search. So the AI either loads everything (slow, bloated) or loads nothing (amnesiac). QMD is the index that makes the whole system actually work at scale.

## The Decay System (How Memory Stays Fresh)

Memory without decay becomes noise. After 90 days, MEMORY.md would be 50,000 words of stale context if you didn't manage it.

The decay rules:

| Category | Window | Behavior |
|---|---|---|
| Hot facts | Accessed in last 7 days | Feature prominently in working memory |
| Warm facts | 8-30 days | Include but deprioritize |
| Cold facts | 30+ days | Drop from working memory, stay in knowledge graph |

Cold facts don't disappear — they just stop being loaded into every session. They're still there when you search for them. They just don't crowd out the things that matter today.

The system tracks `lastAccessed` and `accessCount` on every knowledge graph fact. When the AI uses a fact, it bumps the counter. Weekly memory consolidation re-ranks what's hot.

## The Compound Effect

Here's what this actually does over time.

Week 1: The AI knows your name and tech stack. Saves you 5 minutes of re-explanation per session.

Week 4: The AI knows your active projects, your standing preferences, your recent decisions. It gives advice that fits your situation, not some generic best practice. Saves you 20 minutes of context-setting per session.

Week 12: The AI knows your customer segments, your competitors, your team's skill gaps, your runway, your launch timeline. It operates like a colleague who's been embedded in your business for three months. It surfaces relevant context before you ask. It flags inconsistencies between what you said last week and what you're saying now.

This is not a magic trick. This is what happens when you treat information systematically instead of letting it evaporate at the end of every chat window.

## The Setup (Practical)

If you want to build this:

1. **Create MEMORY.md** in your workspace. Start with 50 lines. What's your current focus? What are your active projects? What are two or three hard preferences the AI should always know?

2. **Set up daily notes** with a cron job or agent log. The format doesn't matter as much as the habit. A 6-line daily entry beats a perfectly structured empty template.

3. **Start a knowledge graph** with a single entity — yourself. One JSON file with your name, your tech stack, your timezone, your communication style. It grows from there.

4. **Wire in a search tool** before you need it. By day 30, you'll have enough memory that unindexed search becomes painful. Do it early.

5. **Write the decay rules down** so you actually follow them. Without explicit rules, working memory bloats. With rules, it stays useful.

---

## The One Mistake That Kills This System

Treating memory like logging instead of like curation.

If you log everything without pruning, you get a haystack. The AI can't find the needle. Every session starts with 10,000 lines of context, most of it stale.

Memory has to be maintained. Not constantly — a 20-minute weekly review is enough. Promote new priorities. Supersede old facts. Archive cold context. The system should feel lighter after a week of use, not heavier.

The goal isn't total recall. The goal is relevant recall. Everything the AI needs, nothing it doesn't.

## Closing

Your AI is only as good as the context it has access to. That context doesn't come from a bigger model. It comes from a memory architecture that accumulates deliberately.

Three layers. One search index. A decay system that keeps it fresh.

Build it once. The compounding starts immediately.
